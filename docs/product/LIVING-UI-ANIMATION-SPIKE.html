Living-UI Animation Spike

Active R&D direction for Celstate motion assets: matted motion assets from our own difference-matting image pipeline — animated as stepped sprite cells, rigged deformations, or procedurally-driven stills — consumed by a React Native runtime. Replaces the video-source + alpha-matting line, whose full record is archived in TRANSPARENT-ANIMATION-RD-SPIKE.md. Read that archive's §0 for why we pivoted; this doc is where we are going.

Plain Current State

  • Phase Hypothesis Direction chosen and reasoned. No sprite sheet has yet been generated or measured. Nothing here is validated.
  • Core bet Matte one generated image with the existing white/black difference matte, then make it move — by stepping sprite cells (Path A), or by animating a single still in the runtime (procedural Path C / rigged Path B, §6). Alpha, cost, and cross-platform consumption collapse to solved or near-solved; the open work is coherence and the runtime.
  • Platform React Native (iOS + Android). Chosen because RN has no robust transparent-video primitive, which makes a PNG/WebP sprite sheet the natural — not the compromise — choice.
  • Key unknown Temporal coherence of generated frames (per-cell "boil"). The resolution ceiling is now bounded — both candidate models render a native 4K square (§3.8) — so the live unknown is coherence plus the crisp-cell floor: how much per-cell detail survives once N frames share the canvas. Both are cheap to test against the pipeline today.
  • Next decision Run the two-phase spike (§7): can we generate usable sheets (G-gate), and can RN consume them at density and right-size them (C-gate)?

1. Vision

Every app looks alike now: sterile, flat, modern, dead. Celstate makes apps feel alive. Not "another pretty image" — motion assets that inhabit the interface and are dropped straight into a running app.

  • A button whose overgrown bushes sway in the wind.
  • A slider that rides railroad tracks as you drag it.
  • A background that teems with life — drifting fireflies, swaying grass, falling leaves — instead of a static gradient.

The asset class we serve, stated precisely, is A* = { stylized, mostly-opaque objects in motion, with bounded soft edges }. Soft volumetrics (smoke, glow, mist, fine hair) are corner cases, explicitly out of MVP scope — see archive §0.1. This narrowing is not a retreat; it is the unlock. It removes the only regime where alpha recovery is genuinely hard.

2. Strategic Thesis

Three pillars, each chosen to fix a specific failure of the archived video path.

PillarClaimFixes
Motion = sprite sheets An animation is an R×C grid of frames in one image; play by stepping cells. Optionally an animated WebP/APNG for the no-interaction case. Cross-platform consumption, looping, interactivity, density.
Alpha = our difference matte Generate the sheet once on white and once on black; recover exact alpha with the pipeline we already ship. No chroma key, no MatAnyone2, no spill repair, no watermark. The entire alpha-recovery problem and the watermark blocker.
Moat = the runtime Generation is a commodity input. Defensibility lives in the consumption layer: a drop-in RN component library + atlas runtime that right-sizes, performs at density, loops, and themes. The "no moat in owning a generator" problem. 
M ≠ f(generator)            # the model that draws the frames is not the product
M  = f(runtime, curation)   # the thing that makes a generated asset trivially usable, is

What "commodity" does and doesn't mean. The base model is a commodity — we don't build it and we swap it freely. The craft of getting coherent, registration-clean, loop-clean frames out of it (reference-image anchoring, prompt structure, loop authoring) is not commodity; it is one of the two hard problems — the other being the runtime — and a sanctioned workstream. "No moat in the generator" means we don't depend on owning the model, not that generation is easy.

Likewise the runtime is open design space. The component names in §5 (<LivingButton>, …) are illustrative, not a spec. How the library is factored, what its API is, and where it draws the seam between asset and motion are the decisions where the product is won — deliberately left to whoever builds it.

3. The Method, Formally

English is lossy here; the symbols are the contract.

3.1 Difference matting on a single sheet

Generate the same content on white (B=255) and black (B=0). Per the live pipeline, alpha is exact:

C_w = F·α + 255·(1−α)
C_b = F·α
  ⇒  α = 1 − (C_w − C_b)/255      (per channel; α = max over channels)
  ⇒  F = C_b / α                  (α > 0)

Applied to one sheet image, this yields clean alpha for every frame at once — provided the white and black passes register sub-pixel. That registration is neither free nor automatic. Because the Gemini 3.x image family is autoregressive and exposes no seed (§3.8), the pipeline gets it the only way available: the black pass is generated img2img, conditioned on the white pass as a reference image (live code: blackSession.sendMessageWithImages(buildBlackBgPrompt(), [whiteImage])). For a single subject this holds well. For an R×C sheet it is the central unproven step — every one of the N cells must reproduce pixel-aligned on black, and a few px of drift in any one cell turns that cell's matte into edge fringe. Per-cell white↔black registration is therefore a first-class G-gate measurement (§7), distinct from temporal coherence (§3.6).

3.2 Sheet geometry

Sheet S: canvas W×H, grid R rows × C cols, N = R·C frames
cell size        (w, h) = (⌊W/C⌋, ⌊H/R⌋)
frame f → grid   (row, col) = (⌊f/C⌋, f mod C),   f ∈ {0 … N−1}
frame f → pixel  (x, y) = (col·w, row·h)

3.3 The fidelity × frame-count ceiling (the central constraint)

For a fixed canvas, frames and per-frame resolution trade off directly:

per-frame area  a(N) = w·h = (W·H) / N        # double the frames ⇒ half the cell
crisp cell needs (w ≥ w_min, h ≥ h_min):
  N_max = ⌊W / w_min⌋ · ⌊H / h_min⌋

Couple this with right-sizing (§3.6): w_min = d_max · ρ_max, where d_max is the largest on-screen size in dp and ρ_max ≈ 3 is the max device pixel ratio. So the model's max output resolution and the asset's display size jointly cap the frame count:

N_max ≈ (W_model · H_model) / (d_max · ρ_max)²      [square cells]

Pixel ceiling (June 2026): both candidate models — Gemini 3.1 Flash Image and Gemini 3.0 Pro Image — can render a native 4K square (4096×4096 = 16.78 MP), not an upscale, so the pixels for the 4K column below exist. One caveat to verify first: gemini-3.1-flash-image-preview has a reported bug where it ignores imageConfig.imageSize and returns ~1K regardless — so guaranteed 2K/4K output may require 3.0 Pro Image until confirmed fixed (§3.8). Model and canvas-shape rationale: §3.8.

AssetDisplayCell @ ρ=3N_max @ 2K (2048²)N_max @ 4K (4096²)
Small accent (icon)48 dp144 px≈ 202≈ 784
Button overlay120 dp360 px≈ 32≈ 121
Hero / background256 dp768 px≈ 7≈ 25

Design rule that falls out of the math: small accents get many frames per sheet (smooth); large hero/background assets get few frames per sheet → tile across multiple sheets, lower the frame rate, or move to a non-sheet path (§6). The table above is the pixel ceiling — necessary, not sufficient.

The binding ceiling is coherence, not pixels. Practitioner reports on this exact model family converge on a hard limit well below the pixel budget: keep a sheet to roughly one subject, 3–6 cells per row, 2–3 rows — order N ≈ 6–18 coherent cells before identity drifts. So although 4K supplies the pixels for ~784 small-accent cells, the model holds only ~12 before boil and per-cell drift dominate. N is coherence-bound — roughly an order of magnitude under the pixel ceiling. This is the single most important number to measure first, and it is what routes anything needing many frames to Paths B/C (§6) rather than to bigger sheets.

3.4 Cost model (why "per-frame" is no longer cost-prohibitive)

C_per_frame    = 2N model passes     # generate each frame independently
C_single_sheet = 2  model passes     # one sheet, matted once
speedup = C_per_frame / C_single_sheet = N

The old objection ("a transparent still per frame is cost-prohibitive") assumed 2N. Single-sheet generation makes cost O(1) in frame count.

3.5 Playback & looping

frame(t) = ⌊t · fps⌋ mod N
loop period  T = N / fps
seamless loop ⇔ d(S_{N−1}, S_0) ≤ ε_loop     # perceptual continuity at the seam

3.6 Coherence ("boil") — the key empirical risk

Independent frames jitter. Even within one sheet, cells can drift. Define the residual not explained by the intended motion field Φ_f:

B = (1/|interior|) · Σ_x ‖ S_{f+1}(x) − warp(S_f ; Φ_f)(x) ‖
accept ⇔ B ≤ B_max

Ambient motion (sway, pulse, drift) is the most forgiving coherence case — and it is exactly our vision. A swaying bush tolerates small B; a character performing a complex action would not. We are lucky here, but B must be measured, not assumed.

3.7 Right-sizing

display size d (dp), device pixel ratio ρ = PixelRatio.get()   # 2 or 3 on phones
required native px  p ≥ d · ρ
render scale        s = (d · ρ) / p
crisp ⇔ s ≤ 1        (s > 1 ⇒ upscale blur)

Raster, so not vector-perfect at arbitrary upscale — but far better than video: author the cell at d_max · ρ_max and it scales down cleanly everywhere. Ship @2x/@3x where the size range is wide.

3.8 Model & canvas-shape decision

Two questions were open when this doc opened: which image model, and what sheet aspect ratio. Both now have answers grounded in the June 2026 specs of the two models we use.

Gemini 3.1 Flash Image in pipelineGemini 3.0 Pro Image
Native max4096² — 16.78 MP4096² — 16.78 MP
Size tiers512p / 1K / 2K / 4K — but preview reportedly ignores imageSize, returning ~1K ⚠1K / 2K / 4K — honoured
Aspect ratios1:1 … 21:9 plus 4:1 & 8:1 (and 1:4, 1:8 @ 512p)1:1 … 21:9 only — no ultra-wide strip
Seed / reproducibilityNone — autoregressive; anchor via reference imageNone — autoregressive; anchor via reference image
≈ cost / image~$0.05 std → ~$0.15 @ 4K~$0.13 (1K/2K) → ~$0.24 (4K)
Edge it buysSpeed, cost, free (R,C) sweeps; wide stripsReliable hi-res + strongest cross-cell identity (the boil lever, §3.6)

Seal on the seed question. Neither model exposes a seed — the whole Gemini 3.x image family is autoregressive (LLM-style), not diffusion, so there is no noise vector to pin. Cross-cell and white↔black consistency therefore come only from reference-image anchoring (img2img), which the pipeline already uses (§3.1). Practical consequence: identity is a prompt/conditioning problem, not a seed-control one, and "60–95% consistent" is the realistic band to design around — not bit-exact reproduction.

Model decision — probe on Flash, escalate to 3.0 Pro for resolution and identity. Sweep (R,C) on Flash 3.1 first: it is already the pipeline's model (GENERATION_CONFIG.model), cheap (~$0.10–$0.30 per two-pass sheet), and fast enough to iterate freely. But verify it actually emits the requested size — if the imageSize bug is live, Flash caps at ~1K and the 4K column is unreachable on it. Move an asset to 3.0 Pro Image when it needs guaranteed 2K/4K or when cross-cell identity is marginal (boil near B_max) — which, given the ~6–18-cell coherence ceiling (§3.3), is most hero/large assets. The split maps onto the pivot tree (§6, §8): Flash is the cheap probe; 3.0 Pro is the upgrade you try before abandoning Path A. Pro's missing 8:1 costs nothing — we want square sheets regardless (below).

Aspect-ratio decision — keep the sheet square, not wide. The instinct toward an 8:1 canvas is the wrong lever for a grid. At a fixed 4K tier the pixel budget is maximised at 1:1 (4096² = 16.78 MP) and collapses as the canvas stretches (8:1 = 4096×512 = 2.1 MP — one-eighth the pixels to subdivide). The sheet's only job is to hold the most pixels; the cell's aspect — which should match the asset — is set by choosing R and C, not by the sheet outline.

sheet aspect      ≈ 1:1             # maximise total pixels to subdivide
cell aspect       = asset aspect    # achieved via R, C — not the sheet outline
choose R, C   s.t. ⌊W/C⌋ : ⌊H/R⌋ ≈ asset aspect,   R·C ≥ N_min(asset)

The ultra-wide ratios (4:1, 8:1 — a Flash-only capability) earn their place only for a literal single-row film strip (a 1×N small accent) or a natively wide single asset (a banner), never for a 2-D grid: at the same cell size — i.e. the same per-frame crispness — a square 4K sheet holds ≈ 8× the frames of an 8:1 strip, exactly the 16.78 MP ÷ 2.1 MP pixel-budget ratio.

Spec provenance — verified 2026-06-13; both models are preview Native 4K (4096²) and the Flash-only 4:1/8:1 ratios: Gemini 3.1 Flash Image model card + Vertex AI model doc. Gemini 3.0 Pro Image 1K/2K/4K tiers and 21:9 ceiling: Vertex AI listings. No-seed (autoregressive) and the Flash imageSize-ignored bug: Google parameter docs + googleapis/js-genai #1461 — the latter is filed against @google/genai, our exact client, so it is the first thing to retest. Re-confirm ceilings, ratios, and the bug before any production cutover.

4. Why React Native Makes This the Obvious Primitive

Choosing RN as the focus is not incidental — it strengthens the thesis.

  • RN has no robust transparent-video primitive. react-native-video can play HEVC-alpha on iOS but Android transparent-video support is poor and inconsistent. The cross-platform codec pain that plagues web video (archive §0.5) is worse on RN. So video is not even a real option here.
  • Alpha images are first-class. PNG and WebP with alpha render natively in <Image> on both platforms. A sprite sheet is just an image. No codec matrix, no fallbacks, no iOS multi-<video> crashes.
  • The runtime is the moat, and RN gives us the surface for it. A small set of components (<LivingButton>, <LivingBackground>, <LivingSlider>) backed by an atlas runtime is exactly the defensible consumption layer §2 calls for.

5. React Native Architecture

Three rendering tiers, ranked by where each wins. All consume the same matted sheet.

TierTechBest forWhy
0 — Simplest expo-image plays an animated WebP / APNG Set-and-forget loops, low instance counts Zero stepping logic; the runtime decodes the loop. Trade-off: no per-frame interactivity/state control.
1 — Interactive <Image> + clipping View + react-native-reanimated Buttons, sliders, state-driven motion Frame index driven on the UI thread (60fps, off JS). Hover/press can change speed, direction, or which frame-range plays.
2 — Density @shopify/react-native-skia Atlas / drawAtlas "Teeming" backgrounds: tens–hundreds of instances One GPU texture, batched per-sprite transforms (RSXform). The right primitive for many simultaneous elements without N draws.

5.1 Tier 1 — interactive frame-stepper (sketch)

// SpriteSheet.tsx — step one cell of an R×C atlas, UI-thread driven.
import { Image } from 'expo-image';
import Animated, {
  useSharedValue, useAnimatedStyle, withRepeat, withTiming, Easing,
} from 'react-native-reanimated';

function SpriteSheet({ source, W, H, rows, cols, fps, cell }) {
  const N = rows * cols;
  const f = useSharedValue(0);                       // frame index (UI thread)
  useEffect(() => {
    f.value = withRepeat(
      withTiming(N, { duration: (N / fps) * 1000, easing: Easing.steps(N, true) }),
      -1, false);                                    // loop forever
  }, [N, fps]);
  const style = useAnimatedStyle(() => {
    'worklet';
    const i = Math.floor(f.value) % N;
    const col = i % cols, row = Math.floor(i / cols);
    return { transform: [{ translateX: -col * cell.w }, { translateY: -row * cell.h }] };
  });
  return (
    <View style={{ width: cell.w, height: cell.h, overflow: 'hidden' }}>
      <Animated.View style={style}>
        <Image source={source} style={{ width: W, height: H }} contentFit="cover" />
      </Animated.View>
    </View>
  );
}

5.2 Tier 2 — density via Skia atlas (sketch, illustrative API)

// Teeming.tsx — K instances from ONE texture, GPU-batched.
import { Canvas, Atlas, useImage, rect, useRSXformBuffer } from '@shopify/react-native-skia';

function Teeming({ sheetUri, cell, cols, rows, count, frame, X, Y }) {
  const tex = useImage(sheetUri);
  const sprites = useMemo(() => Array.from({ length: count }, () => {
    const i = frame.value % (cols * rows);
    return rect((i % cols) * cell.w, Math.floor(i / cols) * cell.h, cell.w, cell.h);
  }), [count, frame]);
  const transforms = useRSXformBuffer(count, (xf, i) => {
    'worklet'; xf.set(/*scale*/ 1, /*radians*/ 0, /*tx*/ X[i], /*ty*/ Y[i]);
  });
  if (!tex) return null;
  return <Canvas style={{ flex: 1 }}><Atlas image={tex} sprites={sprites} transforms={transforms} /></Canvas>;
}

5.3 Format & performance budget

  • Format: WebP-alpha for sheets (smaller than PNG, alpha-correct on both platforms); PNG for tiny hero accents; AVIF to evaluate. Animated WebP / APNG for Tier 0.
  • Never drive frames with setInterval — JS-thread timers jank under load. Use Reanimated (UI thread) or Skia (GPU).
  • GPU memory ≈ W·H·4 bytes per decoded sheet (uncompressed in texture memory, regardless of file size). Budget it; pack/atlas; downscale-on-load to the largest size actually shown.
  • Right-sizing: author at d_max · ρ_max; let RN scale down. Use PixelRatio.get() to pick a variant.

6. Three Paths to Living Motion

One instinct — a matted asset that moves — has three realisations. They are complementary, not competing, and they split the asset range cleanly. Crucially, A is not the default with B/C as fallbacks: because N is coherence-bound to ~6–18 cells (§3.3), the many-frame regime belongs to C and the large-asset regime to B from the start.

A. Generated Sprite Sheet

The model draws all N cells; we matte once and step them. Fastest to prove. Risk: coherence B, per-cell registration (§3.1), and the ~6–18-cell ceiling. Best for short ambient loops on small/medium assets that fit inside the coherent-cell budget.

C. Generate-One + Procedural Motion

Model draws one matted still; a GPU shader animates it — displacement / flow / noise fields (Skia, Reanimated, or a fragment shader). Zero boil, unlimited frames, sub-100 KB, runtime-themeable. For the actual vision — sway, drift, shimmer, fireflies — this is often the primary technique, not a fallback: ambient motion is a continuous field, and a generated sprite grid is only a lossy sampling of it.

B. Generate-One + Rigged Deformation

Model draws one gorgeous asset; a runtime mesh/skeleton animates it — Rive / Lottie / Skia mesh, Spine/Live2D-style. Zero boil, vector-like right-sizing, tiny files. Best for hero/background and large assets, and for structured motion (a slider riding rails) where a deformation field beats sampled pixels.

A is the cheap empirical probe and the only path that needs many coherent generated frames. C and B move the motion out of generated pixels and into the runtime — exactly where §2 says the value lives. All three reuse the difference-matting pipeline on a single sheet or still; none touches a commodity video model. The decision tree (§8) routes each asset to the cheapest path that clears its quality bar.

7. The Spike Plan

Cheap, two-phase, runnable against the live pipeline today. Three target assets exercise the real range: (1) swaying-bush button overlay, (2) slider-on-rails, (3) teeming-background element (drifting fireflies / swaying grass).

Phase G — Generation (does our pipeline produce usable sheets?)

  • Prompt the model for an R×C sprite sheet of each asset; sweep (R,C) from inside the coherent-cell budget (~6–18) outward to find where it breaks. First verify the requested size is honoured — Flash 3.1 may silently return ~1K (§3.8); fall back to 3.0 Pro Image if so.
  • Run the existing white/black pipeline: white pass, then black as img2img anchored on white (§3.1).
  • Measure per-cell white↔black registration, per-frame fidelity, coherence B (§3.6), loop seam ε_loop, and file size.
G-gate (pass requires all):
  per-frame fidelity      ≥ 4/5   founder review, on ≥ 2 of 3 assets
  per-cell registration   white↔black ≤ 1 px drift on every cell   (§3.1 — else matte fringes)
  coherence               B ≤ B_max        (ambient motion reads as coherent, no boil)
  loop seam               d(S_{N−1}, S_0) ≤ ε_loop
  usable frames at size   N ≥ N_min(asset),  N within the coherent-cell budget (§3.3)
  reachable resolution    requested imageSize honoured; crisp-cell px recorded (§3.8)

Phase C — Consumption (can RN use them at density & right-size them?)

  • Expo harness rendering each asset via Tier 1 (Image+Reanimated) and Tier 2 (Skia atlas).
  • Measure on real iOS + Android devices.
C-gate (pass requires all):
  framerate    fps ≥ 58  at K instances, K ∈ {1, 10, 50}, both platforms
  memory       within device budget at K = 50
  right-size   crisp (s ≤ 1) at ρ ∈ {1, 2, 3}
  loop         seamless; no visible seam or hitch
  interactivity state change (press → speed/range) lands within 1 frame

Phase R — Rigged sibling (optional, run if G-gate is marginal on large assets)

Take one generated asset; rig it (Rive/Lottie/Skia mesh); compare coherence, right-sizing, and file size against the sheet for the hero/background case.

8. Decision Gates & Pivot Tree

Go

G-gate and C-gate pass for ≥ 2 of 3 assets. Build the runtime component library; that is the product.

Coherence fails (B too high)

Sprite generation boils or cells drift. Route to Path C (procedural) for ambient motion and Path B (rigged) for hero/structured motion: generate one matted still, animate it in the runtime. Sheets retained only for the small/ambient cases that passed.

Both fail

Generation can't yet make coherent living assets at fidelity. Park the motion bet; the transparent-still product still stands. Re-probe when the image model improves. A respectable outcome, not a failure.

Note the asymmetry with the archived path: here a failure has cheap, already-specified fallbacks (Paths B and C) that still reuse our IP. We are not betting the company on a single unproven mechanism.

9. MVP — Definition & Objective Measures

The gates in §7–8 prove the mechanism. This section defines the product: when have we built an MVP of "apps feel alive," covering the element/background range minus the soft-volumetric corner case? Three things must be measurable — coverage, aliveness, and the gate constants the spike otherwise leaves as bare symbols.

9.1 Coverage — the tickable definition of "everything except corner cases"

MVP = the runtime + a starter asset set that clears the per-class bar on the elements below. Soft volumetrics (smoke, glow, mist, fine hair) are explicitly excluded (§10).

Element classExampleLikely path (§6)In MVP?
Small accent / iconswaying leaf, pulsing dotA or CYes
Interactive controlslider-on-rails, toggleB (rigged)Yes
Button overlayswaying-bush buttonA or CYes
Background / ambient fieldfireflies, grass, leavesC (procedural)Yes
Loader / state feedbackliving spinner, success bloomA or CYes
Soft volumetricsmoke, glow, mist, hair—Corner case

Coverage bar: MVP ships when ≥ 4 of the 5 in-scope classes have at least one asset that passes both gates and renders through a runtime component, on both platforms. That is the falsifiable line for "covers the range minus corner cases."

9.2 Aliveness — the one outcome metric that isn't taste

The vision is a perception claim ("alive, not dead"), so its top-line metric must be a perception test, not founder review. Forced-choice, blind:

2AFC: N ≥ 30 viewers, each shown the living vs static build of the same screen
aliveness ⇔ living preferred as "more alive / more premium" at p < 0.05
            (binomial vs 50%; target effect ≥ 70/30)
guardrail  ⇔ no rise in self-reported distraction / discomfort vs static

Founder review (≥ 4/5) stays as the fast inner-loop filter; the 2AFC is the gate that says the motion earns its place rather than merely renders.

9.3 Calibrating the gate constants (objective in substance, not just form)

B_max, ε_loop, and N_min are symbols until pinned. Each earns a number by calibration against human judgement, not by guess:

ConstantMeaningHow it gets a number
B_maxmax residual before "boil" is visible (§3.6)B of the worst asset a blind panel still rates "no visible boil" vs a clean reference
ε_loopmax seam discontinuity for a seamless loop (§3.5)seam d(S_{N−1},S_0) at which < 1 in 10 viewers can locate the loop point
N_minmin frames for motion to read as smooth, per assetlowest N at fixed fps where the panel stops reporting "steppy"; expect ~8–12 for ambient

Calibrating these on the three spike assets converts every G-gate row from "objective in form" to "objective in substance," and they then hold for every asset after. Note the numbers interlock: N_min (≈ 8–12) sits just under the coherence ceiling (~6–18, §3.3) — confirmation that Path A's usable window is real but narrow, and that anything above it belongs to Paths B/C.

10. Out of Scope (Explicit)

  • Soft volumetrics — smoke, glow, mist, fine hair. The corner case (archive §0.1); not an MVP requirement.
  • Transparent video as a delivery format — cross-platform-hostile, and worse on RN than web.
  • Owning / depending on a commodity generator (Wan-Alpha, Veo, Runway, …) as the source of defensibility. Use whatever is cheapest to make frames; the moat is the runtime.
  • The Alpha Compiler as a primary alpha source. Kept only as an optional raster edge-refinement pass if ever needed (archive §0.7).
  • Web-first. We have chosen RN as the focus. Web is a later port, not the proving ground.

11. Open Questions

  • Reproducibility / seed — resolved. Neither model exposes a seed; the Gemini 3.x image family is autoregressive, not diffusion. Consistency comes only from reference-image anchoring, which the pipeline already does (white → black img2img, §3.1). Implication: cross-cell identity is a prompt/conditioning problem, not a seed-pinning one; design for the 60–95% band, not bit-exactness.
  • Model resolution ceiling — bounded, with a caveat (§3.8). Both Gemini 3.1 Flash Image and Gemini 3.0 Pro Image can render a native 4K square (4096² = 16.78 MP) on Vertex. Caveat to confirm first: gemini-3.1-flash-image-preview reportedly ignores imageConfig.imageSize and returns ~1K — so reaching 2K/4K may require 3.0 Pro Image until fixed. The live question is no longer the ceiling but how many of those pixels survive as coherent cells (§3.3) — order ~12, not ~784.
  • Coherence on ambient motion. Does single-sheet generation hold identity across cells well enough that B ≤ B_max without per-frame conditioning? If not, does img2img/reference conditioning between cells fix it, or do we move to Paths B/C?
  • Loop authoring. Can we reliably prompt cyclic motion so S_{N−1} → S_0 is seamless, or do we need a synthesis step (ping-pong, cross-fade, learned loop)?
  • Theming. Re-generate per palette, or tint at runtime? Living UI implies assets that match the host app's theme.
  • File-size budget. A hi-fidelity multi-frame WebP sheet vs the <200 KB mobile guideline — where does each asset class land?
  • Density ceiling. How many Skia-atlas instances before the C-gate framerate breaks on a mid-tier Android device?
One-line summary for future-us The hard, valuable problem is no longer "make alpha from video." It is "generate coherent, right-sizeable living assets and make them trivially consumable in a React Native app." Alpha is solved by our existing difference matte on a single sheet; the bet is coherence + the runtime.
