Living-UI Animation Spike

Active R&D direction for Celstate living UI: agent-installable component bundles made from generated transparent art layers, strict semantic manifests, and deterministic runtime motion. Initial platform: Expo / React Native / TypeScript. Replaces the video-source + alpha-matting line, whose full record is archived in TRANSPARENT-ANIMATION-RD-SPIKE.md. Read that archive's §0 for why we pivoted; this doc is where we are going.

Plain Current State

  • Phase Build + first generation evidence The runtime control spine is built model-free and hardened: <CelstateLivingButton> and <CelstateLivingSlider> ship as agent-installable bundles over @celstate/living-ui-runtime. The §3.0 behaviour invariants (real press compression κ_press = 0.92 < 1, value-exact thumb, hold/cancel/release, deterministic theming, the typed manifest contract) are machine-proven and measured on a real GPU surface via the web-runtime harness (≈60 fps, κ_press = 0.92 observed, 0 px thumb error).
  • Generation evidence G-gate decomposition — first positive The skin-layer probe (2026-06-15, Gemini 3 Pro Image, prod difference-matte; §3.10) generated isolated transparent skin layers — foliage, seed-pod, soft bloom — that matted clean, isolated, at 0 px white↔black registration, premium quality, on first drafts. The vision's central unknown — can generation decompose the rig's organic slots? — has its first strong yes.
  • Core bet Generation owns the organic / decorative transparent layers (foliage, accents, ambient, blooms — proven viable). The runtime owns all behaviour and the structural chrome (button face, rail, thumb — simple gradient shapes themed by tokens, §9.5). The button-face matte refusing to isolate (§3.10) confirms the split: do not generate flat structural elements.
  • Platform Initial MVP target: Expo React Native + TypeScript. The user is often an LLM coding agent; the deliverable must be a third-party-installable bundle with code, assets, manifest, and QA evidence.
  • Key unknown No longer "feels alive" (model-free, high-confidence) or "can generation decompose at all" (first positive, §3.10). The live unknowns narrow to: (1) black-pass / matte reliability at scale (≈1/4 raw misses — prod background-purity validation + retry must lift it to ≥ 4/5), (2) theming / host-fit and (3) kit coherence, and the human F-gate — does living actually beat static with real viewers.
  • Next decision Close the loop: wire generated matted layers into the rig (replace the hand-authored stand-in) so the real product is visible — generated transparent art skinning a runtime-owned control. Then harden black-pass reliability through the prod pipeline (§3.10) and run the F-gate 2AFC (§9.3) on the Expo harness.

1. Vision

Every app looks alike now: sterile, flat, modern, dead. Celstate makes apps feel alive. Not "another pretty image" and not "a video with alpha" — agent-installable UI components that behave like tactile game UI.

  • An AI coding agent asks for a button; Celstate returns a React Native component bundle whose body presses, holds, cancels, releases, and rebounds, while overgrown foliage breathes around it.
  • An AI coding agent asks for a slider; Celstate returns a component whose thumb position is a function of value, with rails, moss, particles, and haptics-friendly timing supplied by the runtime.
  • An AI coding agent asks for atmosphere; Celstate returns a Skia/Reanimated field of fireflies, grass, leaves, shimmer, or other bounded effects that can be installed into a third-party app.

The MVP product class is not merely A* = { stylized, mostly-opaque objects in motion, with bounded soft edges }. It is: 

Bundle E = (C, M, A, R, Q)
C = typed component code installable by an agent into a third-party app
M = semantic manifest: primitive, states, props, layer roles, dimensions, timing
A = generated transparent art layers / skins / bounded sprite effects
R = deterministic runtime motion mapping UI state → visual state
Q = machine + human QA evidence for alpha, behaviour, installability, and aliveness

Soft volumetrics (smoke, glow clouds, mist, fine hair) are corner cases, explicitly out of MVP scope — see archive §0.1. The first MVP may be narrower still: Expo React Native + TypeScript, one or two primitives, and a strict generated bundle contract.

2. Strategic Thesis

Four pillars, each chosen to fix a specific failure of the archived video path and the first generated-sheet capability result.

PillarClaimFixes
Motion = semantic runtime primitive A control's behaviour is a state machine / continuous mapping, not a bag of frames. The runtime owns press, hold, cancel, release, value, loading, success, and disabled states. The "12 pretty frames but no UI behaviour" failure (§3.9).
Alpha = our difference matte Generate layers/effects/sheets once on white and once on black; recover exact alpha with the pipeline we already ship. No chroma key, no MatAnyone2, no spill repair, no watermark. The entire alpha-recovery problem and the watermark blocker.
Moat = the generation→rig pipeline We don't own the base model, but "generation is the commodity, the runtime is the moat" is backwards: the runtime (Reanimated/Skia/Rive) is standard RN technique a studio reproduces in a quarter. The scarce, defensible link is the craft of getting themeable, decomposable, art-directed-coherent layers out of the model and into rigs. The moat is the integrated pipeline; its least-commodity part is the generation craft, not the runtime. Both the "no moat in owning a generator" problem and the inverse error of trusting the runtime to defend by itself.
User = coding agent The output must be installable by an LLM agent: code, assets, manifest, README, dependency list, test IDs, and QA report. A naked image or APNG is insufficient. The "looks nice in a folder but cannot be used by a third party" failure mode. 
Wrong product:  prompt → finished animated asset
Right product:  prompt → (parts, manifest, component, runtime, QA)

UI state u(t)   ∈ U                 # pressed, held, cancelled, value∈[0,1], loading, success...
Runtime R       : U × time × theme → visual transform field
Generated art A : style/theme/content; never the sole owner of interaction semantics
Render          = compose(A, R(u(t), t, theme))

What "commodity" does and doesn't mean. The base model is a commodity — we don't build it and we swap it freely. What is not commodity is the craft of getting registration-clean, decomposable, themeable, art-directed-coherent layers out of it (reference-image anchoring, prompt structure, layer decomposition, loop authoring) and feeding them into rigs. The runtime is largely standard RN technique — do not mistake it for the moat. "No moat in the generator" means we don't depend on owning the model; it does not mean generation is easy, nor that the runtime defends us by itself.

The runtime is no longer open-ended prose. The MVP runtime is a product surface: an Expo RN package plus generated component bundles. The component names in §5 (<LivingButton>, <LivingSlider>, …) are now the natural first API shape, even if their exact props change.

3. The Method, Formally

English is lossy here; the symbols are the contract.

3.0 UI behaviour is a function, not a frame sequence

The central correction after the first capability test: a UI control is not primarily an animation. It is a stateful function with visual output.

Primitive P      = (U, Props, Theme, A, R)
U_button         = {idle, hover?, pressed, held, cancelled, disabled, loading, success}
U_slider         = {value ∈ [0,1], dragging ∈ Bool, disabled ∈ Bool, focused ∈ Bool}
A                = generated transparent layers and bounded effects
R_P              : U × Props × Theme × t → transforms, filters, particles, frame ranges
visual(t)        = compose(A, R_P(u(t), props, theme, t))

Button invariant:   bodyCenter(passive frames) ≈ constant
Button press:       height(pressed) / height(idle) ≤ κ_press   where κ_press < 1
Slider invariant:   thumbX(value) = lerp(trackStart, trackEnd, value)
Agent contract:     install(E, app) ⇒ component renders + responds without manual asset surgery

A sprite sequence can encode a sampled path through U. It does not own U. Any implementation that cannot answer "what happens when the user holds, cancels, drags halfway, changes theme, or changes label?" is not a living UI primitive.

3.1 Difference matting on a single sheet

Generate the same content on white (B=255) and black (B=0). Per the live pipeline, alpha is exact:

C_w = F·α + 255·(1−α)
C_b = F·α
  ⇒  α = 1 − (C_w − C_b)/255      (per channel; α = max over channels)
  ⇒  F = C_b / α                  (α > 0)

Applied to one sheet image or one generated layer, this yields clean alpha — provided the white and black passes register sub-pixel. That registration is neither free nor automatic. Because the Gemini 3.x image family is autoregressive and exposes no seed (§3.8), the pipeline gets it the only way available: the black pass is generated img2img, conditioned on the white pass as a reference image (live code: blackSession.sendMessageWithImages(buildBlackBgPrompt(), [whiteImage])). For a single subject or generated layer this holds well. For an R×C sheet every one of the N cells must reproduce pixel-aligned on black, and a few px of drift in any one cell turns that cell's matte into edge fringe. Per-cell white↔black registration is therefore a first-class G-gate measurement (§7), distinct from temporal coherence (§3.6).

3.2 Sheet geometry

Sheet S: canvas W×H, grid R rows × C cols, N = R·C frames
cell size        (w, h) = (⌊W/C⌋, ⌊H/R⌋)
frame f → grid   (row, col) = (⌊f/C⌋, f mod C),   f ∈ {0 … N−1}
frame f → pixel  (x, y) = (col·w, row·h)

3.3 The fidelity × frame-count ceiling (the central constraint)

For a fixed canvas, frames and per-frame resolution trade off directly:

per-frame area  a(N) = w·h = (W·H) / N        # double the frames ⇒ half the cell
crisp cell needs (w ≥ w_min, h ≥ h_min):
  N_max = ⌊W / w_min⌋ · ⌊H / h_min⌋

Couple this with right-sizing (§3.6): w_min = d_max · ρ_max, where d_max is the largest on-screen size in dp and ρ_max ≈ 3 is the max device pixel ratio. So the model's max output resolution and the asset's display size jointly cap the frame count:

N_max ≈ (W_model · H_model) / (d_max · ρ_max)²      [square cells]

Pixel ceiling (June 2026): both candidate models — Gemini 3.1 Flash Image and Gemini 3.0 Pro Image — can render a native 4K square (4096×4096 = 16.78 MP), not an upscale, so the pixels for the 4K column below exist. One caveat to verify first: gemini-3.1-flash-image-preview has a reported bug where it ignores imageConfig.imageSize and returns ~1K regardless — so guaranteed 2K/4K output may require 3.0 Pro Image until confirmed fixed (§3.8). Model and canvas-shape rationale: §3.8.

AssetDisplayCell @ ρ=3N_max @ 2K (2048²)N_max @ 4K (4096²)
Small accent (icon)48 dp144 px≈ 202≈ 784
Button overlay120 dp360 px≈ 32≈ 121
Hero / background256 dp768 px≈ 7≈ 25

Design rule that falls out of the math: small accents get many frames per sheet (smooth); large hero/background assets get few frames per sheet → tile across multiple sheets, lower the frame rate, or move to a non-sheet path (§6). The table above is the pixel ceiling — necessary, not sufficient.

The binding ceiling is coherence, not pixels. Practitioner reports on this exact model family converge on a hard limit well below the pixel budget: keep a sheet to roughly one subject, 3–6 cells per row, 2–3 rows — order N ≈ 6–18 coherent cells before identity drifts. So although 4K supplies the pixels for ~784 small-accent cells, the model holds only ~12 before boil and per-cell drift dominate. N is coherence-bound — roughly an order of magnitude under the pixel ceiling. This is the single most important number to measure first, and it is what routes anything needing many frames to Paths B/C (§6) rather than to bigger sheets.

3.4 Cost model (why "per-frame" is no longer cost-prohibitive)

C_per_frame    = 2N model passes     # generate each frame independently
C_single_sheet = 2  model passes     # one sheet, matted once
speedup = C_per_frame / C_single_sheet = N

The old objection ("a transparent still per frame is cost-prohibitive") assumed 2N. Single-sheet generation makes cost O(1) in frame count.

3.5 Playback & looping

frame(t) = ⌊t · fps⌋ mod N
loop period  T = N / fps
seamless loop ⇔ d(S_{N−1}, S_0) ≤ ε_loop     # perceptual continuity at the seam

3.6 Coherence ("boil") — the key empirical risk

Independent frames jitter. Even within one sheet, cells can drift. Define the residual not explained by the intended motion field Φ_f:

B = (1/|interior|) · Σ_x ‖ S_{f+1}(x) − warp(S_f ; Φ_f)(x) ‖
accept ⇔ B ≤ B_max

Ambient motion (sway, pulse, drift) is the most forgiving coherence case — and it is exactly our vision. A swaying bush tolerates small B; a character performing a complex action would not. We are lucky here, but B must be measured, not assumed.

3.7 Right-sizing

display size d (dp), device pixel ratio ρ = PixelRatio.get()   # 2 or 3 on phones
required native px  p ≥ d · ρ
render scale        s = (d · ρ) / p
crisp ⇔ s ≤ 1        (s > 1 ⇒ upscale blur)

Raster, so not vector-perfect at arbitrary upscale — but far better than video: author the cell at d_max · ρ_max and it scales down cleanly everywhere. Ship @2x/@3x where the size range is wide.

3.8 Model & canvas-shape decision

Two questions were open when this doc opened: which image model, and what sheet aspect ratio. Both now have answers grounded in the June 2026 specs of the two models we use.

Seal on the seed question. Neither model exposes a seed — the whole Gemini 3.x image family is autoregressive (LLM-style), not diffusion, so there is no noise vector to pin. Cross-cell and white↔black consistency therefore come only from reference-image anchoring (img2img), which the pipeline already uses (§3.1). Practical consequence: identity is a prompt/conditioning problem, not a seed-control one, and "60–95% consistent" is the realistic band to design around — not bit-exact reproduction.

Model decision — probe on Flash, escalate to 3.0 Pro for resolution and identity. Sweep (R,C) on Flash 3.1 first: it is already the pipeline's model (GENERATION_CONFIG.model), cheap (~$0.10–$0.30 per two-pass sheet), and fast enough to iterate freely. But verify it actually emits the requested size — if the imageSize bug is live, Flash caps at ~1K and the 4K column is unreachable on it. Move an asset to 3.0 Pro Image when it needs guaranteed 2K/4K or when cross-cell identity is marginal (boil near B_max) — which, given the ~6–18-cell coherence ceiling (§3.3), is most hero/large assets. The split maps onto the pivot tree (§6, §8): Flash is the cheap probe; 3.0 Pro is the upgrade you try before abandoning Path A. Pro's missing 8:1 costs nothing — we want square sheets regardless (below).

Aspect-ratio decision — keep the sheet square, not wide. The instinct toward an 8:1 canvas is the wrong lever for a grid. At a fixed 4K tier the pixel budget is maximised at 1:1 (4096² = 16.78 MP) and collapses as the canvas stretches (8:1 = 4096×512 = 2.1 MP — one-eighth the pixels to subdivide). The sheet's only job is to hold the most pixels; the cell's aspect — which should match the asset — is set by choosing R and C, not by the sheet outline.

sheet aspect      ≈ 1:1             # maximise total pixels to subdivide
cell aspect       = asset aspect    # achieved via R, C — not the sheet outline
choose R, C   s.t. ⌊W/C⌋ : ⌊H/R⌋ ≈ asset aspect,   R·C ≥ N_min(asset)

The ultra-wide ratios (4:1, 8:1 — a Flash-only capability) earn their place only for a literal single-row film strip (a 1×N small accent) or a natively wide single asset (a banner), never for a 2-D grid: at the same cell size — i.e. the same per-frame crispness — a square 4K sheet holds ≈ 8× the frames of an 8:1 strip, exactly the 16.78 MP ÷ 2.1 MP pixel-budget ratio.

Spec provenance — verified 2026-06-13; both models are preview Native 4K (4096²) and the Flash-only 4:1/8:1 ratios: Gemini 3.1 Flash Image model card + Vertex AI model doc. Gemini 3.0 Pro Image 1K/2K/4K tiers and 21:9 ceiling: Vertex AI listings. No-seed (autoregressive) and the Flash imageSize-ignored bug: Google parameter docs + googleapis/js-genai #1461 — the latter is filed against @google/genai, our exact client, so it is the first thing to retest. Re-confirm ceilings, ratios, and the bug before any production cutover.

4. Why React Native Makes This the Obvious Primitive

Choosing RN as the first target is not incidental — it strengthens the thesis and gives coding agents a narrow, testable install target.

  • RN has no robust transparent-video primitive. react-native-video can play HEVC-alpha on iOS but Android transparent-video support is poor and inconsistent. The cross-platform codec pain that plagues web video (archive §0.5) is worse on RN. So video is not even a real option here.
  • Alpha images are first-class. PNG and WebP with alpha render natively in <Image> on both platforms. A sprite sheet is just an image. No codec matrix, no fallbacks, no iOS multi-<video> crashes.
  • RN gives the runtime its surface. A small set of components (<LivingButton>, <LivingSlider>, <LivingToggle>, <LivingField>) backed by Reanimated, Skia Atlas, and optionally Rive rigs is the consumption layer. It is necessary but not sufficient for defensibility: the moat is the generation→rig pipeline as a whole (§2), of which this runtime is the standard, reproducible half.
  • Agents can install it. Expo RN projects have predictable dependency installation, TypeScript imports, file placement, and emulator/device verification. That makes "third-party usable" falsifiable.

5.3 Tier 2 — density via Skia atlas (sketch, illustrative API)

// Teeming.tsx — K instances from ONE texture, GPU-batched.
import { Canvas, Atlas, useImage, rect, useRSXformBuffer } from '@shopify/react-native-skia';

function Teeming({ sheetUri, cell, cols, rows, count, frame, X, Y }) {
  const tex = useImage(sheetUri);
  const sprites = useMemo(() => Array.from({ length: count }, () => {
    const i = frame.value % (cols * rows);
    return rect((i % cols) * cell.w, Math.floor(i / cols) * cell.h, cell.w, cell.h);
  }), [count, frame]);
  const transforms = useRSXformBuffer(count, (xf, i) => {
    'worklet'; xf.set(/*scale*/ 1, /*radians*/ 0, /*tx*/ X[i], /*ty*/ Y[i]);
  });
  if (!tex) return null;
  return <Canvas style={{ flex: 1 }}><Atlas image={tex} sprites={sprites} transforms={transforms} /></Canvas>;
}

5.4 Format & performance budget

  • Format: WebP-alpha for sheets (smaller than PNG, alpha-correct on both platforms); PNG for tiny hero accents; AVIF to evaluate. Animated WebP / APNG for Tier 0.
  • Never drive frames with setInterval — JS-thread timers jank under load. Use Reanimated (UI thread) or Skia (GPU).
  • GPU memory ≈ W·H·4 bytes per decoded sheet (uncompressed in texture memory, regardless of file size). Budget it; pack/atlas; downscale-on-load to the largest size actually shown.
  • Right-sizing: author at d_max · ρ_max; let RN scale down. Use PixelRatio.get() to pick a variant.
Runtime-doc provenance — checked 2026-06-14 Expo documents Reanimated as the included path for smooth RN animation, Reanimated exposes animated styles driven by shared values, React Native Skia's Atlas is built for many transformed instances of the same texture, and Rive's current RN runtime supports view-model/state-machine style data binding. See official docs before cutting a production dependency set.

6. Three Paths to Living Motion

One instinct — generated transparent art that makes the app feel alive — has three realisations. They split the asset range cleanly, and the routing is now explicit: controls belong to authored runtime motion (C or B), ambient to C, and A to bounded effects only. A is not a default with B/C as fallbacks — because generated sheets are coherence-bound and not reliably state-semantic (§3.9), controls are runtime-owned from the start. The decisive principle: author the motion first; let generation only skin it. That keeps generation reliability off the critical path of feel — what the vision lives or dies on — while preserving infinite visual variety.

C. Generate-One + Procedural Motion controls + ambient

Model draws one or more matted layers; runtime code (Reanimated/Skia) animates them — transforms, springs, displacement, flow/noise fields, particles. Zero generated-frame boil, unlimited frames, naturally coherent, runtime-themeable. Primary path for ambient life and for most tactile controls: the motion is authored, the art is generated.

B. Authored Rig + Generated Skin high-craft controls

A hand-authored mesh/skeleton/state machine owns behaviour — Rive / Skia mesh, Spine/Live2D-style; generation supplies the skin and variants that fill the rig's defined slots. Zero boil, strong interaction semantics, small files, and — decisively — feel guaranteed independent of generation. For controls whose deformation is too rich for plain procedural motion. The rig is finite and excellent; the skins are infinite.

A. Generated Sprite Sheet bounded effects only

The model draws all N cells; we matte once and step them. Use only for short bounded loops: success blooms, sparkles, leaf wiggles, tiny loaders, firefly wing flaps. Risks: coherence B, per-cell registration (§3.1), layout obedience, semantic drift. Never the path for button presses, sliders, or toggles.

C and B keep motion in the runtime and off the critical path of feel — exactly where §2 says the value lives. A is the cheap empirical probe and an export target for effects only. All three reuse the difference-matting pipeline on sheets, stills, or layers; none touches a commodity video model. The decision tree (§8) routes each primitive to the lowest-risk path that clears its behaviour bar — for controls, never A.

7. The Spike Plan

Original plan: cheap two-phase generation + consumption spike. Current plan after §3.9: prove feel before generation. The phases are ordered to fail cheap — the model-free aliveness proof (Phase F) is the kill-fast gate; generation (Phase G) is wired in only once the rig and the feel are proven; installability is sequenced last because it is known-solvable engineering, not a strategic risk.

Phase F — Feel proof (model-free; the kill-fast gate)

  • Hand-author one <LivingButton> in Reanimated/Skia — the MVP default rig; Rive only if Path C can't clear the 2AFC on a high-craft control — with hand-made art, no generation in the loop. Make it as alive and tactile as craft allows.
  • Run the 2AFC aliveness test (§9.3) against the static equivalent.
  • This is the cheapest decisive test we have: a hand-crafted living control either beats static or the vision is dead. Spend days, not weeks, before committing to generation.
F-gate (pass requires all):
  aliveness     living preferred at p < 0.05, effect ≥ 70/30   (§9.3)
  feel          founder review ≥ 4/5 on press/hold/cancel/release
  model-free    achieved with zero generated art     # proves feel is ours to control
Fail ⇒ the vision is not ready; park the motion bet before any generation spend.

Phase P — Primitive generalisation (does one rig shape span the types?)

  • Extend the proven authored approach to <LivingSlider> and one ambient field (Path C), still with hand-made art.
  • Confirm one runtime/rig shape covers press-controls, value-controls, and ambient without bespoke per-component re-engineering.
  • Measure press latency, hold/cancel correctness, slider value correctness, disabled/loading states, accessibility labels/roles, and frame rate.
P-gate (pass requires all):
  button semantics    idle, press, hold, cancel, release, disabled, loading all observable
  slider semantics    thumbX(value) monotonic and exact within 1 px at sampled values
  latency             interaction visual response ≤ 1 frame
  fps                 ≥ 58 on representative iOS + Android devices
  abstraction         one rig/runtime shape spans control + ambient, no bespoke rebuild

Phase G — Generation as constrained skinner (can it feed the proven rig?)

Generation fills defined slots in an existing rig (skins, palettes, variants); it never owns behaviour and never decomposes an arbitrary control unaided.

Status (2026-06-15): the decomposition row was pulled forward as a standalone probe (§3.10), ahead of the F-gate, to de-risk the central capability question directly — and it returned a strong positive for organic slots (0 px registration, premium first-drafts). What remains of Phase G: lift black-pass reliability to ≥ 4/5, then theming and kit coherence.

  • For controls, generate the rig's slot art to a fixed spec — body skin, rail, thumb, foliage front/back, particles, masks — not free-form. Do not ask the model to own press/drag semantics.
  • For bounded effects (Path A), prompt an R×C sheet; sweep (R,C) from inside the coherent-cell budget (~6–18) outward. First verify the requested size is honoured — Flash 3.1 may silently return ~1K (§3.8); fall back to 3.0 Pro Image if so.
  • Run the existing white/black pipeline: white pass, then black as img2img anchored on white (§3.1).
  • Measure the three real unknowns — decomposition reliability, theming / host-fit, cross-component coherence — alongside registration, alpha, file size, and (sheets only) coherence B and loop seam ε_loop.
G-gate (pass requires all):
  decomposition       clean separable slot layers on ≥ 4/5 runs, no manual surgery
  theming / host-fit  same rig reskins to ≥ 3 distinct host palettes and reads native
  kit coherence       button + slider + ambient generate as one art-directed world
  registration        white↔black ≤ 1 px drift per cell/layer   (§3.1 — else fringes)
  fidelity            ≥ 4/5 founder review, on ≥ 2 of 3 assets
  bounded effects:    B ≤ B_max,  loop seam ≤ ε_loop,  N within coherent budget (§3.3)
  resolution          requested imageSize honoured; crisp-cell px recorded (§3.8)

Phase C — Consumption & install (density, right-size, agent-installable)

  • Expo harness rendering each asset via Tier 1 (Image+Reanimated) and Tier 2 (Skia atlas).
  • Verify an agent installs the bundle into a blank third-party Expo app with no manual asset surgery (§9.2). Sequenced last: this is engineering, not a strategic unknown.
  • Measure on real iOS + Android devices.
C-gate (pass requires all):
  framerate      fps ≥ 58  at K instances, K ∈ {1, 10, 50}, both platforms
  memory         within device budget at K = 50
  right-size     crisp (s ≤ 1) at ρ ∈ {1, 2, 3}
  loop           seamless; no visible seam or hitch
  interactivity  state change (press → speed/range) lands within 1 frame
  installability blank Expo app + generated README succeeds, no bespoke human help

8. Decision Gates & Pivot Tree

Go

F-gate proves feel model-free; P/G/C-gates then pass for <LivingButton> and <LivingSlider>, with at least one generated-skin bundle installed into a blank Expo app by an agent. Build the rig library + generation feeder; that is the product.

Feel fails model-free (F-gate)

If a hand-rigged control can't beat static in the 2AFC, the vision is not ready — and no amount of generation fixes it. Park the motion bet cheaply, before any generation spend. The transparent-still product still stands.

Generation can't feed the rig (G-gate)

Feel is proven but generation won't reliably decompose, theme, or stay coherent across a kit. Fall back to authored / curated skins for MVP and mature generation behind the proven runtime — the rigs still ship. Generated sheets stay bounded-effects only (§3.9).

Note the asymmetry with the archived path: failures are cheap and ordered. F-gate can kill the whole vision in days, not a quarter; a G-gate failure falls back to authored skins without losing the rig runtime. We are not betting the company on generated sprite sheets — or on generation at all for the feel.

9. MVP — Definition & Objective Measures

The gates in §7–8 prove the mechanism. This section defines the product: when have we built an MVP of "apps feel alive" that an AI coding agent can actually install into a third-party app? Five things must be settled — installability, coverage, aliveness, theming, and the gate constants the spike otherwise leaves as bare symbols.

9.1 Coverage — the tickable definition of "everything except corner cases"

MVP = the runtime + generated bundle pipeline + starter asset/component set that clears the per-class bar on the elements below. Soft volumetrics (smoke, glow clouds, mist, fine hair) are explicitly excluded (§10).

Element classExampleLikely path (§6)In MVP?
Button primitivepress/hold/cancel/release with living surroundC (B if needed)Yes — first proof
Slider primitivethumb rides rails as value changesC (B if needed)Yes — first proof
Small accent / iconswaying leaf, pulsing dotA or CYes
Background / ambient fieldfireflies, grass, leavesCAfter controls
Loader / state feedbackliving spinner, success bloomA or CAfter controls
Soft volumetricsmoke, glow, mist, hair—Corner case

The MVP, landed. A Celstate runtime package plus two agent-installable bundles — <LivingButton> and <LivingSlider> — for Expo RN + TypeScript, with motion authored in Reanimated/Skia (Rive reserved for a high-craft escalation), art delivered as the §5.1 bundle (parts + manifest + README + QA), and theming via manifest tokens (§9.5). It ships only when, on both iOS and Android: F-gate proves feel model-free, then the two bundles — generated skin where it clears G-gate, authored skin where it doesn't — pass P/G/C-gates in a blank Expo app installed by an agent. Broader coverage (≥ 4 of 5 classes) is the next milestone, not the first MVP.

9.2 Installability — the agent-facing product metric

Install test:
  Input:  blank Expo RN TS app A0, generated bundle E, README instructions I
  Agent:  generic coding agent with shell access
  Pass:   agent applies I, app builds, component renders, state props work

No manual asset surgery, no hidden local paths, no bespoke human explanation.

Installability is the product boundary that defines "done": our users are often LLM coding agents, so the bundle must be explicit enough for an agent to execute. But it is a definitional boundary, not a strategic risk — it is known-solvable engineering (peer deps + manifest + README), which is why §7 sequences it last, after feel and generation are proven.

9.3 Aliveness — the one outcome metric that isn't taste

The vision is a perception claim ("alive, not dead"), so its top-line metric must be a perception test, not founder review. Forced-choice, blind:

2AFC: N ≥ 30 viewers, each shown the living vs static build of the same screen
aliveness ⇔ living preferred as "more alive / more premium" at p < 0.05
            (binomial vs 50%; target effect ≥ 70/30)
guardrail  ⇔ no rise in self-reported distraction / discomfort vs static

Founder review (≥ 4/5) stays as the fast inner-loop filter; the 2AFC is the gate that says the motion earns its place rather than merely renders.

9.4 Calibrating the gate constants (objective in substance, not just form)

B_max, ε_loop, and N_min are symbols until pinned. Each earns a number by calibration against human judgement, not by guess:

ConstantMeaningHow it gets a number
B_maxmax residual before "boil" is visible (§3.6)B of the worst asset a blind panel still rates "no visible boil" vs a clean reference
ε_loopmax seam discontinuity for a seamless loop (§3.5)seam d(S_{N−1},S_0) at which < 1 in 10 viewers can locate the loop point
N_minmin frames for motion to read as smooth, per assetlowest N at fixed fps where the panel stops reporting "steppy"; expect ~8–12 for ambient

Calibrating these on bounded-effect assets converts every G-gate row from "objective in form" to "objective in substance." Structured controls additionally need P-gate constants: press response latency, compression ratio, cancellation correctness, slider value error, and accessibility correctness.

9.5 Theming — the decided MVP mechanism (was an open fork)

The vision is the host app feeling alive, so theming is required, not optional. The decision: manifest theme tokens applied by the runtime — no model call at install. Art ships in two layer roles: structural layers (form, shadow, geometry) palette-neutral and used as-is; tintable layers (foliage, accent, glow) authored neutral/grayscale-with-alpha so the runtime multiplies the host token colour through them. Re-generation per palette is deferred post-MVP; runtime tint without the neutral-authoring discipline is rejected (hue-shifting baked-in colour degrades). This is exactly what the G-gate "reskins to ≥ 3 host palettes" row measures.

10. Out of Scope (Explicit)

  • Generating structural control chrome. The button face, slider rail, and thumb are flat gradient shapes — the difference-matte can't isolate a frame-filling flat element (§3.10), and theme tokens reskin them for free. Draw them in the runtime; spend generation only on organic / decorative layers.
  • Over-investing in hand-authored art. Hand-made art is the F-gate stunt double — built "good enough to test feel," then replaced. Polishing it past that bar is wasted motion; the shippable art is generated (§3.10).
  • Generated sprite sheets owning UI behaviour (press, drag, toggle, cancel). Proven unreliable (§3.9, κ_press = 0.998); behaviour is runtime-owned, always (§6, Path C/B). Sheets stay for bounded effects only.
  • Continuous soft volumetrics — smoke, mist, fine hair (archive §0.1); not an MVP requirement. Note: bounded soft effects (a success bloom) matte cleanly with partial alpha (§3.10) and are in scope via Path A.
  • Transparent video as a delivery format — cross-platform-hostile, and worse on RN than web.
  • Owning / depending on a commodity generator (Wan-Alpha, Veo, Runway, …) as the source of defensibility. Use whatever is cheapest to make frames; the moat is the generation→rig pipeline (§2), not any one model.
  • The Alpha Compiler as a primary alpha source. Kept only as an optional raster edge-refinement pass if ever needed (archive §0.7).
  • Web-first. We have chosen RN as the focus. Web is a later port, not the proving ground.
  • Arbitrary-framework output. MVP is allowed to be Expo RN only. Cross-framework generation comes after one runtime has real proof.

11. Open Questions

Triaged so the MVP is genuinely landed: none of these blocks the control MVP build. Each is already resolved, empirical (a gate measures it — no decision to make), or scoped to a later milestone.

  • Reproducibility / seed — resolved. Neither model exposes a seed; the Gemini 3.x image family is autoregressive, not diffusion. Consistency comes only from reference-image anchoring, which the pipeline already does (white → black img2img, §3.1). Implication: cross-cell identity is a prompt/conditioning problem, not a seed-pinning one; design for the 60–95% band, not bit-exactness.
  • Model resolution ceiling — bounded, with a caveat (§3.8). Both Gemini 3.1 Flash Image and Gemini 3.0 Pro Image can render a native 4K square (4096² = 16.78 MP) on Vertex. Caveat to confirm first: gemini-3.1-flash-image-preview reportedly ignores imageConfig.imageSize and returns ~1K — so reaching 2K/4K may require 3.0 Pro Image until fixed. The live question is no longer the ceiling but how many of those pixels survive as coherent cells (§3.3) — order ~12, not ~784.
  • Coherence on ambient motion. Does single-sheet generation hold identity across cells well enough that B ≤ B_max without per-frame conditioning? If not, does img2img/reference conditioning between cells fix it, or do we move to Paths B/C?
  • Loop authoring. Can we reliably prompt cyclic motion so S_{N−1} → S_0 is seamless, or do we need a synthesis step (ping-pong, cross-fade, learned loop)?
  • Layer decomposition — first positive evidence (§3.10), now a reliability question. Generation isolates the rig's organic slots cleanly at 0 px registration. The open part is no longer "can it" but "how reliably": the raw black-pass miss rate (~1/4) must drop under background-purity validation + retry to clear the G-gate's ≥ 4/5 bar.
  • Theming / host-fit & kit coherence — still open G-gates. Does the same rig reskin to ≥ 3 host palettes and read native, and do button + slider + ambient generate as one art-directed world? These, not "does it feel alive," are what remain to decide the vision.
  • File-size budget. A hi-fidelity multi-frame WebP sheet vs the <200 KB mobile guideline — where does each asset class land?
  • Density ceiling. How many Skia-atlas instances before the C-gate framerate breaks on a mid-tier Android device?
  • Agent install ceiling. How much dependency setup and native configuration can a generic coding agent handle without user intervention?
One-line summary for future-us Alpha is solved; decomposition of organic layers is now shown to work (§3.10). The hard, valuable problems left are generation reliability (black-pass purity, theming, kit coherence) and the one human gate — does living actually beat static in the 2AFC. Feel and "can the model make the art" are no longer the central risks.

12. Durable Lessons

This section is intentionally redundant. Future agents should be able to recover the enduring decision context from this file alone.

LessonDurable Form
Generated full-control sprite sheets are not the high-confidence path. They can be beautiful and matte-clean while failing the UI contract. A button press, slider drag, toggle snap, or cancel gesture is a semantic runtime behaviour.
Alpha and organic decomposition are strengths, not the product. The 2026-06-14 sheet matted clean; the 2026-06-15 skin probe (§3.10) isolated foliage / seed / soft-bloom layers at 0 px registration, premium, first-draft. "Can the model make decomposable transparent art?" is answered yes. The differentiator is up-stack: reliability, theming, coherence, and runtime behaviour.
Generate organic; draw structural. Generation excels at organic / decorative transparent layers (foliage, accents, blooms). Structural control chrome (button face, rail, thumb) is a flat shape the matte can't isolate (§3.10) and tokens reskin for free — the runtime draws it. Routing art to the wrong half wastes spend and fails the matte.
An opaque matte is a black-pass failure, not an art failure. ~1/4 of raw generations leave a non-black background on the black pass, so the difference-matte sees no difference and returns fully opaque — while the subject art is perfect. Gate with background-purity validation (config.blackBgMaxMean) + retry; never read an opaque matte as "the model can't make the art."
Hand-authored art is the F-gate stunt double, never the product. Hand art exists only to test whether living motion is preferred at all — build it "good enough to test," then replace it with generated layers (§3.10). Polishing the stand-in past that bar is the easiest way to waste a week.
Model self-report is not evidence. The model claimed 4×3, no grid lines, and idle/press/hold/release. The image was 3×4, had asymmetric grid contamination, and did not compress. Always inspect pixels and behaviour metrics.
Sprite sheets remain useful. Use them for bounded loops and effects: blooms, sparkles, loaders, small ambient cycles. Do not make them own arbitrary UI state.
The MVP user is often an AI coding agent. Return an explicit bundle: component code, assets, manifest, README, dependencies, QA report, and test hooks. A folder of images is not enough.
Quality bar: video-game feel or do not ship. If runtime-authored controls still feel mediocre, park the motion product. Do not ship weak "living UI" that undermines the brand.
Prove feel before generation. Making a control feel alive is model-free and high-confidence. Reliably generating themeable, decomposable, coherent art to feed it is the real risk. Sequence so a hand-rigged 2AFC validates or kills the vision first, for days of spend — not after a generation pipeline is built.
The moat is the pipeline, not the runtime. Reanimated / Skia / Rive are standard RN technique a studio reproduces in a quarter. The defensible, least-commodity link is the generation→rig craft: decomposition, theming, and art-direction coherence. Spend confidence-building effort there.
