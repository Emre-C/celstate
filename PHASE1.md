# Phase 1 — Native Transparent Image Generation

## Premise

We have unlimited engineering capacity. Every line of code is generated by frontier-class models that exceed human engineering capability. There is no "we'll do the simple version first" — we do the correct version, the only version, from the start. Every seam is tight. Every edge case is handled. Every failure mode has a recovery path.

---

## The Core Invention

No existing consumer tool generates images with native transparency. Every competitor follows the same broken pipeline: generate opaque → remove background with ML → pray the edges survive. The ML step is lossy, unpredictable, and destroys semi-transparent elements (glass, smoke, hair, shadows).

Celstate eliminates that entire pipeline with **difference matting** — a deterministic, mathematically exact technique that recovers perfect alpha channels without any ML post-processing.

### How It Works

We generate the **same subject twice** — once on pure white, once on pure black — using Gemini's multi-turn chat to guarantee subject consistency. Then we solve for alpha and foreground color per-pixel using closed-form equations.

**The math is exact, not approximate:**

Given a foreground pixel `F` with alpha `α`, composited over background `B`:

```
C = F·α + B·(1 - α)
```

On white (`B = 255`): `C_w = F·α + 255·(1 - α)`
On black (`B = 0`):   `C_b = F·α`

Solving:

```
α = 1 - (C_w - C_b) / 255        // per-channel, take max for robustness
F = C_b / α                       // recover true foreground color (when α > 0)
```

This is not a heuristic. This is the compositing equation solved in reverse. It handles:
- **Fully opaque pixels**: `C_w ≈ C_b` → `α ≈ 1` ✓
- **Fully transparent pixels**: `C_w ≈ 255, C_b ≈ 0` → `α ≈ 0` ✓
- **Semi-transparent pixels** (glass, smoke, thin hair): fractional `α` recovered exactly ✓
- **Colored semi-transparency**: foreground color recovered correctly from `C_b / α` ✓

No ML background remover on Earth can match this on semi-transparent elements. This is our moat.

### Dimension Mismatch Handling

Gemini does not guarantee pixel-identical dimensions across turns, even in the same chat. We do not hope. We enforce.

1. **Request identical `imageConfig`** on both turns (same `aspectRatio`, same `imageSize`).
2. **After decoding**, compare dimensions. If they differ:
   - Resize the larger image down to match the smaller using Lanczos resampling (highest quality downscale).
   - Log the mismatch for prompt tuning telemetry.
3. **If aspect ratios differ** (not just a few pixels off, but fundamentally different compositions): this is a generation failure. Retry from Turn 1.

### Background Purity Validation

We do not trust that Gemini produced a pure white or pure black background. We verify.

**After each generation pass**, sample the corner regions (4 corners, 32×32 pixel patches) and compute:
- Mean channel values
- Standard deviation

**For white-bg pass**: Mean must be > 245 per channel, stddev < 5.
**For black-bg pass**: Mean must be < 10 per channel, stddev < 5.

If validation fails, retry that pass with a reinforced prompt (e.g., "The background MUST be pure white #FFFFFF with absolutely no texture, shadow, or gradient. The entire background area must be a single flat color."). Maximum 2 retries per pass before failing the generation.

## Module Design

### `src/convex/lib/gemini.ts` — Gemini Client

Single responsibility: manage the Gemini API connection, handle chat sessions, return structured results.

```typescript
// Types
interface GeminiImageResult {
  imageBase64: string;
  mimeType: "image/png" | "image/jpeg";
}

interface GeminiChatSession {
  sendMessage(prompt: string): Promise<GeminiImageResult>;
  sendMessageWithImage(prompt: string, image: GeminiImageResult): Promise<GeminiImageResult>;
}

// Public API
function createChatSession(config: {
  model: string;
  aspectRatio: string;
  imageSize: string;
}): GeminiChatSession;
```

**Model selection**: `gemini-3.1-flash-image-preview` — best balance of speed, cost, and quality. The multi-turn chat API is critical for subject consistency.

**Config**: Always request `responseModalities: ['IMAGE']` (no text needed in the response — we want pure image output). Always specify `aspectRatio: "1:1"` and `imageSize: "1K"` as defaults (configurable per-generation if we later expose aspect ratio to users).

### `src/convex/lib/prompts.ts` — Prompt Templates

Isolated from all other logic. This module will be tuned heavily based on empirical results. Every other module treats prompts as opaque strings.

```typescript
function buildWhiteBgPrompt(userPrompt: string): string;
function buildBlackBgPrompt(): string;
function buildWhiteBgRetryPrompt(userPrompt: string): string;
function buildBlackBgRetryPrompt(): string;
```

**White-bg prompt strategy**:
```
{userPrompt}. 

The subject is centered with comfortable margins from all edges. 
The background is pure solid white (#FFFFFF). 
No shadows cast onto the background. No gradients. No textures. 
No ground plane. The subject floats against a perfectly flat white field.
```

**Black-bg prompt strategy** (Turn 2 in the same chat):
```
Now regenerate the exact same subject — identical in every detail: 
same pose, same proportions, same colors, same lighting on the subject, 
same angle, same level of detail. The ONLY change is the background, 
which must now be pure solid black (#000000). No shadows cast onto 
the background. No reflections. No gradients. The subject floats 
against a perfectly flat black field. Everything about the subject 
itself must be pixel-identical to the previous image.
```

**Why we also send the white-bg image as a reference in Turn 2**: The chat history already contains it (as the model's response), but explicitly including it as an input reference in the Turn 2 message gives Gemini a stronger signal for fidelity preservation. The docs confirm up to 14 reference images are supported.

### `src/convex/lib/matte.ts` — Difference Matte Engine

Pure computation. No side effects. No I/O. Takes two raw pixel buffers, returns one.

```typescript
interface MatteInput {
  whiteBg: Uint8ClampedArray;  // RGB or RGBA pixel data, row-major
  blackBg: Uint8ClampedArray;  // RGB or RGBA pixel data, row-major
  width: number;
  height: number;
}

interface MatteOutput {
  pixels: Uint8ClampedArray;   // RGBA pixel data with computed alpha
  width: number;
  height: number;
}

function differenceMatte(input: MatteInput): MatteOutput;
```

**Implementation details:**

Per-pixel (stride of 3 for RGB input, 4 for RGBA):

```
for each pixel (i):
  // Solve alpha from compositing equation
  diffR = whiteR - blackR    // = 255 * (1 - α) per channel
  diffG = whiteG - blackG
  diffB = whiteB - blackB

  // Use max channel difference for most robust alpha estimate
  // (minimizes noise amplification from any single noisy channel)
  maxDiff = max(diffR, diffG, diffB)
  alpha = 255 - maxDiff       // α in [0, 255]

  // Recover foreground color from black-bg image
  // On black bg: C_b = F·α → F = C_b / (α/255)
  if alpha > 0:
    alphaFraction = alpha / 255
    finalR = clamp(blackR / alphaFraction, 0, 255)
    finalG = clamp(blackG / alphaFraction, 0, 255)
    finalB = clamp(blackB / alphaFraction, 0, 255)
  else:
    finalR = finalG = finalB = 0

  output[i*4 + 0] = round(finalR)
  output[i*4 + 1] = round(finalG)
  output[i*4 + 2] = round(finalB)
  output[i*4 + 3] = alpha
```

**Edge refinement**: After the initial matte pass, apply a single cleanup pass:
- Pixels with `alpha < 3` → force to `alpha = 0` (kill near-zero noise)
- Pixels with `alpha > 252` → force to `alpha = 255` (solidify near-opaque)
- This 1% tolerance band eliminates Gemini's compression artifacts at the boundaries without affecting legitimate semi-transparency

**Multi-channel alpha estimation**: We use `max(diffR, diffG, diffB)` rather than averaging because:
- Averaging amplifies noise from channels where the foreground color is close to the background color
- Max gives the most conservative (least transparent) estimate, which produces cleaner edges
- For a pixel that is truly opaque, all three diffs will be ≈ 0, so max = 0, alpha = 255 ✓
- For a pixel that is truly transparent, all three diffs will be ≈ 255, so max = 255, alpha = 0 ✓

### `src/convex/generation.ts` — Orchestrator Action

The only module that touches multiple concerns. It wires together: auth, credits, Gemini, matte, storage.

```typescript
// Convex action (can call external APIs + internal mutations/queries)
export const generate = action({
  args: {
    prompt: v.string(),
  },
  handler: async (ctx, args) => {
    // 1. Auth check (via ctx.auth)
    // 2. Credit check + deduction (internal mutation)
    // 3. Create generation record with status "generating" (internal mutation)
    // 4. Generate white-bg image (Gemini chat Turn 1)
    //    - Validate background purity
    //    - Retry with reinforced prompt if needed (max 2 retries)
    // 5. Generate black-bg image (Gemini chat Turn 2)
    //    - Validate background purity
    //    - Retry with reinforced prompt if needed (max 2 retries)
    // 6. Dimension check + resize if needed
    // 7. Difference matte
    // 8. Encode final RGBA PNG
    // 9. Upload to Convex storage
    // 10. Update generation record: status "complete", attach storage ID
    //
    // On failure at any step:
    // - Refund credits (internal mutation)
    // - Update generation record: status "failed", attach error message
  },
});
```

### `src/convex/schema.ts` — Data Schema

```typescript
import { defineSchema, defineTable } from "convex/server";
import { v } from "convex/values";

export default defineSchema({
  users: defineTable({
    // Managed by Convex Auth
    email: v.string(),
    credits: v.number(),
  }).index("by_email", ["email"]),

  generations: defineTable({
    userId: v.id("users"),
    prompt: v.string(),
    status: v.union(
      v.literal("generating"),
      v.literal("complete"),
      v.literal("failed")
    ),
    resultStorageId: v.optional(v.id("_storage")),
    // Keep intermediate images for debugging/quality analysis
    whiteBgStorageId: v.optional(v.id("_storage")),
    blackBgStorageId: v.optional(v.id("_storage")),
    creditsCost: v.number(),
    aspectRatio: v.string(),
    createdAt: v.number(),
    completedAt: v.optional(v.number()),
    error: v.optional(v.string()),
    // Telemetry
    generationTimeMs: v.optional(v.number()),
    retryCount: v.optional(v.number()),
    dimensionMismatch: v.optional(v.boolean()),
  })
    .index("by_user", ["userId", "createdAt"])
    .index("by_status", ["status"]),
});
```

**Why we store intermediate images** (`whiteBgStorageId`, `blackBgStorageId`):
- Enables quality analysis: if the final matte looks wrong, we can inspect the source pair
- Enables prompt tuning: we can review what Gemini actually produced vs. what we asked for
- Enables future improvements: if we improve the matte algorithm, we can re-process existing pairs without re-generating
- Storage is cheap; re-generation is expensive (API cost + latency)
- These are purged on the same 30-day retention schedule as the final image

## Failure Modes & Recovery

| Failure | Detection | Recovery |
|---|---|---|
| Gemini returns text instead of image | Response has no `inlineData` part | Retry with explicit `responseModalities: ["IMAGE"]` |
| Gemini returns image with impure background | Corner patch validation fails | Retry with reinforced prompt (up to 2x) |
| Gemini returns different dimensions across turns | Post-decode dimension comparison | Lanczos resize to match smaller; log for telemetry |
| Gemini returns fundamentally different subjects | Cannot detect automatically (v1) | Mitigated by chat context + reference image in Turn 2. Future: perceptual hash comparison |
| Gemini rate limit (429) | HTTP status code | Exponential backoff: 1s → 2s → 4s |
| Gemini server error (5xx) | HTTP status code | Retry up to 3x with backoff |
| `sharp` unavailable in Convex runtime | Import/init failure | Fallback: pure TypeScript pixel manipulation with `pngjs` (zero native deps) |
| User double-clicks generate | Optimistic credit deduction is atomic (Convex mutation) | Second call fails credit check cleanly |
| Action crashes mid-generation | Convex action timeout / unhandled exception | Generation stays in "generating" status; scheduled cleanup job marks stale generations as "failed" and refunds credits after 5 minutes |

## Performance Characteristics

| Step | Expected Latency | Notes |
|---|---|---|
| Credit check + deduction | < 50ms | Convex mutation, single document read + write |
| Gemini Turn 1 (white-bg) | 3–8s | API latency for 1K image generation |
| Gemini Turn 2 (black-bg) | 3–8s | Same session, may be slightly faster due to cached context |
| Difference matte (1K×1K) | < 100ms | ~1M pixels × 10 ops/pixel = trivial for modern compute |
| PNG encoding | < 200ms | `sharp` or `pngjs` RGBA encoding |
| Storage upload | < 200ms | Convex file storage |
| **Total** | **7–17s** | Dominated entirely by Gemini API calls |

The matte computation is O(n) where n = pixel count. At 4K (16M pixels), this is still < 500ms. The bottleneck is always the two sequential Gemini calls, which cannot be parallelized (Turn 2 depends on Turn 1's chat context).

## Dependencies

| Package | Purpose | Fallback |
|---|---|---|
| `@google/genai` | Gemini API client (JS SDK) | Direct REST calls to `generativelanguage.googleapis.com` |
| `sharp` | PNG decode/encode, resize | `pngjs` (pure JS, no native bindings) |

Both are minimal, well-maintained, and have no transitive dependency risk. `sharp` is preferred for performance; `pngjs` is the zero-risk fallback if native bindings are problematic in the Convex runtime.
